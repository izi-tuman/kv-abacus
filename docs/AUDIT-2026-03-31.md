## 1. Критические проблемы

### ~~1.1 Context value пересоздаётся на каждый рендер~~ DONE

**Файлы:** `lib/auth-context.tsx`, `lib/settings-context.tsx`

Объект value передаваемый в Provider создаётся заново при каждом рендере. Это вызывает каскадный ре-рендер ВСЕХ потребителей контекста.

**Исправление:** Обернуть value в `useMemo`:
```tsx
const value = useMemo(() => ({
  currentUser, currentRole, setCurrentUser, setCurrentRole, logout,
}), [currentUser, currentRole]);
```

### ~~1.2 Отсутствуют Error Boundaries~~ DONE

**Проблема:** Нет `error.tsx` ни на одном уровне маршрутов. Любая необработанная ошибка приводит к белому экрану.

**Исправление:** Добавить `app/error.tsx` с UI восстановления и кнопкой "Попробовать снова".

### ~~1.3 Race conditions в Promise.all + setState~~ DONE

**Файлы:** страницы History, Rental, BookingCalendar

При размонтировании компонента во время выполнения `Promise.all`, вызывается `setState` на размонтированном компоненте. `mountedRef` используется не везде.

**Исправление:** Добавить AbortController или проверку `mountedRef` во все async-операции.

### ~~1.4 Нет подтверждения удаления на некоторых страницах~~ DONE (не требовалось)

**Статус:** При проверке выяснилось, что `CrudModal` уже содержит `ConfirmDeleteDialog` с полным flow подтверждения. Все три модалки (HouseModal, EquipmentModal, ServiceModal) используют `CrudModal` с `onDelete` и `deleteMessage` — подтверждение удаления работает корректно.

## 2. Архитектурные проблемы

### 2.1 Отсутствие кеширования запросов

Каждый переход между страницами инициирует новые API-запросы. Нет механизма дедупликации или кеширования.

**Рекомендация:** Рассмотреть SWR или TanStack Query для:
- Автоматического кеширования
- Дедупликации запросов
- Stale-while-revalidate
- Автоматических повторных попыток

### 2.2 Отсутствие request timeout

В `lib/api.ts` axios не настроен с timeout. Запросы могут зависать бесконечно на медленной сети.

**Исправление:** Добавить `timeout: 15000` в конфигурацию axios.

### 2.3 Нет loading.tsx / not-found.tsx

Отсутствуют файлы Suspense-границ и 404-обработки на уровне маршрутов Next.js.

### 2.4 settings-context: нет обработки ошибок в updateSettings

`updateSettings()` не имеет try-catch. При ошибке API состояние будет рассинхронизировано с сервером.

### 2.5 equipmentRentalsApi.getRentals() — нет параметра signal

В отличие от `bookingsApi.getBookings()`, метод `getRentals()` не принимает AbortSignal. Невозможно отменить запрос при размонтировании.

## 3. Проблемы производительности

### 3.1 Отсутствие React.memo на карточках списков

**Компоненты без мемоизации:**
- `BookingCard` — получает статические props, ре-рендерится при любом изменении родителя
- `ClientCard`, `HouseCard`, `EquipmentCard`, `ServiceCard`, `UserCard`
- `HistoryCard`, `RentalCard`

**Исправление:** Обернуть в `React.memo()`.

### 3.2 Inline-функции в JSX

**BookingModal.tsx:** `toggleService()`, `handleSubmit()` пересоздаются на каждый рендер.
**ClientPicker.tsx / HousePicker.tsx:** `handleSelect()`, `handleClear()` пересоздаются.
**EquipmentPicker.tsx:** `getQuantity()`, `setQuantity()`, `removeItem()`.

**Исправление:** Обернуть в `useCallback`.

### 3.3 Нет виртуального скроллинга

Длинные списки (клиенты, история бронирований) рендерятся полностью. При большом количестве данных будет тормозить.

**Рекомендация:** Рассмотреть `react-window` или `tanstack-virtual` для списков > 50 элементов.

### 3.4 Множественные API-вызовы при монтировании

`useBookingData` загружает 4 API-эндпоинта при каждом монтировании компонента. Нет кеширования между переходами.

### 3.5 Нет debounce на поиске

Поиск клиентов (ClientsPage) фильтрует на каждый символ без задержки. При большом списке может тормозить.

## 4. UI/UX проблемы

### 4.1 Слишком маленькие touch-target

| Компонент | Размер | Минимум |
|-----------|--------|---------|
| EquipmentPicker +/- кнопки | 28x28px | 44x44px |
| Pagination кнопки страниц | 32x32px | 44x44px |
| BookingCard телефон-иконка | 24x24px | 44x44px |

**Стандарт Apple/Google:** минимум 44x44px для touch-target.

### 4.2 Нет focus trap в кастомных dropdown

`ClientPicker`, `HousePicker` — кастомные выпадающие списки не управляют фокусом. Клавиатурная навигация затруднена.

### 4.3 Drag handle без accessible name

`ModalSheet.tsx` — элемент перетаскивания не имеет aria-label для screen reader.

### 4.4 Скрытие скроллбаров глобально

В `globals.css` скроллбары скрыты через `*` селектор. Это:
- Дорогой селектор (применяется ко всем элементам)
- Скрывает индикацию прокрутки на десктопе
- Лучше применять точечно к конкретным контейнерам

### 4.5 Нет тёмной темы

Цвета определены в OKLCH (что хорошо), но только для светлой темы. Нет поддержки `prefers-color-scheme: dark`.

### 4.6 Страница Settings

- Нет кнопки "Отмена" для сброса формы
- Сообщение "Сохранено" показывается 2 секунды — легко пропустить
- Нет оптимистичного обновления UI

### 4.7 Отсутствие toast-уведомлений

Нет единой системы уведомлений для пользователя. Ошибки и успехи обрабатываются по-разному на каждой странице.

## 6. Проблемы безопасности

### 6.1 localStorage для хранения userId

User ID хранится в localStorage, что уязвимо к XSS-атакам. Рассмотреть HttpOnly cookies.

### 6.2 Нет CSRF-защиты

API-запросы не включают CSRF-токены.

### 6.3 Нет клиентской валидации

Формы полагаются целиком на серверную валидацию. Добавить базовую клиентскую валидацию для быстрой обратной связи.

## 7. Проблемы по страницам

### Главная (BookingCalendar)

- Race condition: AbortController только для бронирований, не для остальных fetch
- `preselectedClient` / `preselectedHouseId` — состояние может утечь между модальными окнами
- Три отдельных состояния модалов не координированы

### Клиенты

- Поиск без debounce
- Нет плавного скролла наверх при пагинации
- Пагинация: кнопки страниц слишком маленькие (32px)

### Прокат оборудования (Rental)

- Загрузка пользователей молча игнорирует ошибки (только console.log)
- Вложенные scroll-контейнеры могут давать плохой UX на мобильных

### История (History)

- `preselectedHouseId` state объявлен но не используется
- Promise.all без очистки при размонтировании
- Ошибки загрузки домов/клиентов/пользователей игнорируются (`catch(() => {})`)

### Отчёты (Reports)

- Небезопасный type casting: `data as FinanceReport[]`
- Нет функции экспорта/скачивания
- Нет disabled-состояния инпутов во время загрузки

### Дома (Houses)

- Нет диалога подтверждения удаления
- Нет сортировки списка

### Настройки (Settings)

- Нет обработки ошибок в updateSettings
- Нет кнопки отмены

### Пользователи (Users)

- Если загрузка ролей упадёт — dropdown будет пустым без ошибки
- Кастомный handleSave обходит стандартный механизм useCrudPage

## 10. Положительные стороны

- **Архитектура компонентов:** Отличная организация по фичам
- **Паттерны модалов:** Консистентное использование ModalSheet и CrudModal
- **TypeScript:** Корректная типизация, нет `any`
- **Хуки:** useCrudPage, useBookingData, usePullToRefresh — хорошо продуманы
- **CSS-переменные в OKLCH:** Современный подход к цветам
- **Мобильные оптимизации:** Предотвращение zoom на iOS, momentum scroll, safe-area padding
- **Accessibility:** aria-labels на навигации и кнопках действий
- **Кодовая база чистая:** Минимальные зависимости, нет лишних пакетов
- **Biome вместо ESLint/Prettier:** Быстрый единый инструмент
